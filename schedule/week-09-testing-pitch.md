## วิธีเขียน Test Case ที่ดี

ใช้ [testing-table template](../worksheets/testing-table.md) — test case ที่ดีต้องครอบคลุม:

1. **กรณีปกติ** — สภาพแวดล้อมธรรมดา ระบบต้องบอก NORMAL
2. **กรณีต้องเตือน** — สร้างเหตุการณ์จริง ระบบต้องเตือน
3. **กรณีก้ำกึ่ง** — ค่าใกล้ threshold พอดี ระบบทำตัวยังไง
4. **กรณีโหด** — ถอดสาย sensor / เปลี่ยนเร็ว ๆ / ใช้งานต่อเนื่อง 5 นาที

ตัวอย่าง:

| Test | Input | Expected Output | Actual Output | Pass/Fail |
|---|---|---|---|---|
| 1 | แสงน้อย | LED ติด | LED ติด | Pass |
| 2 | แสงมาก | LED ดับ | LED ดับ | Pass |
| 3 | เอามือบังครึ่งหนึ่ง | LED ติด | ติดช้า | Partial |

**กติกา: Fail ไม่ใช่เรื่องแย่** — ทีมที่เจอ fail แล้วอธิบายสาเหตุ + แก้ได้ จะได้คะแนนมากกว่าทีมที่ Pass หมดแบบไม่ได้ทดสอบจริง

## 🏆 Team Challenge: "Bug Hunt ข้ามทีม"

จับคู่ทีม (1↔2, 3↔4, ...) แลกกันทดสอบระบบของอีกฝ่าย 10 นาที:
- ทีมผู้ทดสอบพยายาม **ทำให้ระบบเพื่อนพัง/แจ้งเตือนผิด** ด้วยวิธีสร้างสรรค์ (ห้ามทำลายฮาร์ดแวร์)
- ทุก bug ที่หาเจอ = 1 ดาวให้ทีมผู้หา และทีมเจ้าของระบบต้องจดลง testing table

ทีมหา bug เก่งสุดได้รางวัล "QA Engineer ยอดเยี่ยม" — และทุกทีมได้ test case ฟรีกลับบ้าน

## เตรียม Pitch (ใช้ [pitch-template](../worksheets/pitch-template.md))

โครง 5–6 นาที:
1. ปัญหาคืออะไร ใครเดือดร้อน (1 นาที)
2. ระบบทำงานอย่างไร + system diagram (1 นาที)
3. **Demo สด** (2 นาที)
4. ข้อมูล + ผลทดสอบ (1 นาที)
5. ข้อจำกัด + ถ้ามีเวลาต่อจะทำอะไร (1 นาที)
